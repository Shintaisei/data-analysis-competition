# 協調フィルタリング系の改善案と実装

atmaCup #15 1位解法を参考にした**協調フィルタリング・接続情報活用**の改善案と、現状できている実装をまとめる。  
当コンペでは **批評家 = user、映画 = item、「レビューした」= 接続** に対応づける。

---

## 0. 協調フィルタリングの計算の流れ（噛み砕き）

「協調フィルタで何を計算しているか」を、入力から最終的な特徴まで順にまとめる。

### 0.1 入力になるもの

- **使う情報**: 「**誰がどの映画をレビューしたか**」だけ。  
  Fresh か Rotten かは使わない（接続の有無だけ = implicit feedback）。
- **単位**: 1 行 = 1 件のレビュー（批評家 A が映画 M をレビューした、という 1 件）。

### 0.2 行列にする

- train と test の**全レビュー**について、「批評家 ID」と「映画 ID」のペアを集める。
- **批評家 × 映画** の大きな表（行列）を作り、  
  「その批評家がその映画をレビューしていれば **1**、していなければ **0**」を入れる。  
  （実際のコードでは 1 のところだけを記録する疎な行列にしている。）
- この行列が「**誰が何をレビューしたか**」の 0/1 の地図になる。

### 0.3 行列分解（BPR または ALS）でベクトルを作る

- 上の 0/1 行列を **BPR** や **ALS** というアルゴリズムで「分解」する。
- 分解の結果、次の 2 つが得られる：
  - **批評家ベクトル**: 批評家ごとに、長さ 16（または 32 など）の数値のベクトル。
  - **映画ベクトル**: 映画ごとに、同じ長さの数値のベクトル。
- イメージとしては、  
  「**似たレビュー履歴の批評家は似たベクトル**」「**似た批評家にレビューされる映画は似たベクトル**」になるように中身が決まる。
- このベクトルたちが「**協調フィルタの計算結果**」の中身。

### 0.4 各行の特徴としてどう使うか

- 予測したいのは「**この 1 件のレビュー（批評家 A × 映画 M）が Fresh か Rotten か**」。
- そこで、**その行に対応する批評家のベクトル** と **その映画のベクトル** を、その行の特徴として使う。
- **パターン A（今のメイン）**:  
  批評家ベクトル 16 個の数値 + 映画ベクトル 16 個の数値 = **32 列** を、もともとの 55 特徴に**足して**、LightGBM に渡す。  
  → 合計 55 + 32 = 87 特徴で「この組み合わせの傾向」を学習させる。
- **パターン B（内積 1 列）**:  
  批評家ベクトルと映画ベクトルを**内積**（対応する成分のかけ算の和）して **1 つの数** にする。  
  その 1 列だけを 55 特徴に足す。  
  → 「この批評家×この映画の相性スコア」を 1 つの特徴として渡す。

### 0.5 2-hop 集約とは（別の計算）

- 協調フィルタの「行列分解」とは別に、**2-hop** は次のような計算。
- **その映画をレビューした批評家たち**を、train のなかで集める。
- その批評家たちについて、  
  「Fresh になった割合の平均」「critic_te の平均」「レビュー数」などを計算する。
- その 1 つずつを**新しい 1 列**として、その行（その批評家×その映画）に付ける。
- 意味としては「**この映画をレビューした人たちは、全体として Fresh にしやすいか / どんな傾向か**」を数値にしたもの。
- **2-hop の割り算**:  
  「この批評家の critic_te」÷「その映画をレビューした批評家の critic_te の平均」を 1 列にする。  
  1 位で効いた「値 ÷ ユーザー平均」の、映画側バージョン。

### 0.6 まとめ（何がベースで何を足しているか）

- **現在の最高スコア 0.76101** は、  
  **55 特徴（embedding + PCA16 + doc_x_critic_te）** に、  
  **BPR で作った批評家・映画ベクトル 32 列** を足した **87 特徴** で学習した 1 本。
- 2-hop 実験ノート（`train_2hop_experiments.ipynb`）の **ベースはこの 1 本と同じ**。  
  `get_setup(use_best_pipeline=True)` で 55 特徴を用意し、`get_bpr_base(ctx, factors=16)` で同じ BPR 16 の 32 列を足している。  
  したがって「BPR のみ」の実験（`run_experiment(ctx, "bpr_only")`）は、0.76101 を出した提出と**同じ特徴・同じベース**になっている。

## 2. 現状の実装

### 2.1 コード（lib/improvement_candidates.py）

- **`run_atmacup_ratio(ctx, base_feature, suffix)`**  
  - 批評家ごとに train のみで `base_feature` の平均を計算。  
  - 特徴量「値 / 批評家平均」を 1 列追加（0 除算は 1.0、未知批評家は global 平均）。  
  - 55 特徴 + 1 列で LGB 学習し、`submission_atmacup_ratio_{suffix}.csv` を保存。

- **`run_atmacup_implicit(ctx, method, factors, suffix)`**  
  - train+test の (critic, movie) を 0/1 の交互行列にし、`implicit` の ALS または BPR で fit。  
  - `user_factors`（批評家ベクトル）と `item_factors`（映画ベクトル）を各行に付与し、55 特徴に 2×factors 列を追加。  
  - LGB 学習し、`submission_atmacup_implicit_{suffix}.csv` を保存。

### 2.2 実行ノートブック（train_atmacup_5patterns.ipynb）

**ベース**: 0.75493 の土台（55 特徴: embedding + PCA16 + doc_x_critic_te）。この 5 パターンから **implicit BPR 16 で 0.76101**（現時点の最高）を達成。

**5 パターン**（提出ファイル）:

|---|------|----------------|
| 1 | 割り算: genre_Documentary / 批評家平均 | submission_atmacup_ratio_genre_doc.csv |
| 2 | 割り算: critic_name_te_ts / 批評家平均 | submission_atmacup_ratio_critic_te.csv |
| 3 | Implicit ALS factors=16 | submission_atmacup_implicit_als16.csv |
| 4 | Implicit BPR factors=16 | submission_atmacup_implicit_bpr16.csv |
| 5 | 割り算: runtime_bin / 批評家平均 | submission_atmacup_ratio_runtime_bin.csv |

**依存**: `pip install implicit`（requirements.txt に追加済み）。

### 2.3 BPR ベース + 2-hop 実験（lib/two_hop.py / train_2hop_experiments.ipynb）

- **ベース**: **0.76101 を出した提出と同一**。`get_setup(use_best_pipeline=True)` で 55 特徴、`get_bpr_base(ctx, factors=16)` で BPR 16 の 32 列を足す。これは `run_atmacup_implicit(ctx, "bpr", 16, "bpr16")` が内部で使っているのと同じ `_build_implicit_embeddings` を `get_bpr_base` が呼ぶため、同じ 87 特徴になる。したがって「BPR のみ」の実験は最高スコアの土台そのもの。
- **`lib/two_hop.py`**  
  - `add_2hop_features(train_df, test_df, columns)` … 映画ごとに train で集約（Fresh 率平均・critic_te 平均・レビュー数）を計算し、指定列だけ train/test に追加。  
  - `run_experiment(ctx, experiment_name, use_2hop_cols=None)` … BPR ベースで学習し、`use_2hop_cols` で指定した 2-hop 列を追加して `submission_2hop_{experiment_name}.csv` を保存。  
- **ノートブック**: `train_2hop_experiments.ipynb` で「BPR のみ」「BPR + 2-hop 1 列」「BPR + 全2-hop」などを 1 本ずつ実行し、Public スコアの増減を見ながら追加する列を決められる。

| **submission_2hop_bpr32_only** | BPR **32** の批評家・映画ベクトル（計 64 列） | BPR の次元を 16→32 に増やした。表現力が上がるか試す。 |

| **submission_2hop_bpr_2hop_ratio** | BPR 16 の 32 列 **+ 2-hop 割り算 1 列**（critic_te / その映画の批評家の TE 平均） | 「この批評家の傾向」を「その映画をレビューした人たちの傾向」で割った比。1 位で効いた「値÷ユーザー平均」の映画版。 |
| **submission_2hop_bpr32_2hop_ratio** | BPR **32** の 64 列 **+ 2-hop 割り算 1 列** | 上と同じ割り算を、BPR 32 の土台で試した版。 |
| **submission_2hop_bpr_fresh** | BPR 16 の 32 列 **+ 1 列**: その映画をレビューした批評家の **Fresh 率の平均** | 「この映画をレビューした人たちは、何割 Fresh にしたか」を 1 数値で付与。 |
| **submission_2hop_bpr_critic_te** | BPR 16 の 32 列 **+ 1 列**: その映画をレビューした批評家の **critic_te の平均** | 「この映画をレビューした人たちの critic_te（傾向スコア）の平均」を 1 列で付与。 |
| **submission_2hop_bpr_count** | BPR 16 の 32 列 **+ 1 列**: その映画の **レビュー数**（train 内） | 「この映画は何本レビューされたか」を 1 列で付与。 |
| **submission_2hop_bpr_all2hop** | BPR 16 の 32 列 **+ 上 3 つ**（Fresh 率平均・critic_te 平均・レビュー数） | 2-hop の 3 列をまとめて足した版。 |
| **submission_2hop_bpr_all2hop_ratio** | BPR 16 の 32 列 **+ 2-hop 3 列 + 2-hop 割り算 1 列** | 全2-hop に加えて「critic_te / その映画の批評家 TE 平均」も足した版。 |

**用語の整理**

- **BPR**: 「誰がどの映画をレビューしたか」の 0/1 行列を分解して、批評家ごと・映画ごとの**ベクトル**を作る手法。そのベクトルを特徴に使う = 協調フィルタの部分。
- **内積 1 列**: 批評家ベクトルと映画ベクトルをかけ算の和で 1 个数にしたもの。「この組み合わせの相性」を 1 特徴で渡す。
- **2-hop**: 行列分解とは別。「**その映画をレビューした批評家たち**」について、Fresh 率平均・critic_te 平均・人数などを計算し、その行に 1 列ずつ付ける。
- **2-hop 割り算**: 「この批評家の critic_te」÷「その映画をレビューした批評家の critic_te の平均」。傾向が「その映画のレビュワー平均」からどれだけずれているかを 1 列で表す。

**2-hop は今「しっかり使えてる」か**

- ファイル名はすべて `submission_2hop_*` だが、これは**2-hop 用モジュール／ノートから出した提出**という意味で、**中身で 2-hop 特徴を使っているかは別**。
- **2-hop 特徴が実際に使われるのは** `run_experiment(..., use_2hop_cols=...)` で列を指定したときだけ（bpr_fresh, bpr_count, bpr_all2hop など）。
- **現時点の最高（0.76479）の bpr64_only** は `use_2hop_cols=None` で回しており、**2-hop は 1 列も入っていない（BPR 64 のみ）**。軸1の提出も同様に 2-hop なし。
- つまり **「今のベストは 2-hop を使っていない」**。2-hop を足した提出は当コンペではスコアが落ちているため、現状は**意図的に 2-hop を使わない方針**になっている。

**まとめ（意外な結果）**

- **一番効いたのは BPR の次元を増やすこと**。BPR 16 → 32 → 64 と増やすと 0.76101 → 0.76366 → **0.76479** と伸びた。2-hop や内積を足すより「BPR の factors を 64 にする」が最も効いた。
- **2-hop を足すと多くのパターンでスコアが落ちた**。Fresh 率平均・全2-hop・割り算を足すと 0.72 台まで落ちている。当コンペでは 2-hop は過学習や分布ずれを招きやすい可能性がある。
- **内積 1 列**は、BPR ベクトルと一緒に足す（bpr_plus_dot）と 0.760 台でほぼ並ぶが、内積だけ（bpr_dot_only）だと 0.753 で弱い。**単体よりは BPR ベクトルを残して足す方が無難。**
- **今後のベースライン**: 超える目標は **0.76479**（submission_2hop_bpr64_only.csv）。BPR 64 を土台にさらに試すか、2-hop は使わず BPR の次元・正則化などに絞るのがよさそう。

**なぜ 2-hop でスコアが下がるか（想定される要因）**

1. **target リーク（movie_fresh_rate_mean）**  
   `movie_fresh_rate_mean` は「その映画をレビューした行の **target（Fresh/Rotten）の平均**」で計算している。  
   train の**ある 1 行**にこの値を付けるとき、「その映画をレビューした行」の中に**その行自身も含まれる**。  
   → 自分自身の正解ラベルが特徴に混ざっており、**train では擬似的に当たりやすく**、test では同じ映画でも「自分は除いた平均」ではないので汎化せず、**過学習で Public が落ちる**。  
   **対策案**: 映画ごとの平均を取るときに **leave-one-out**（その行を除いて平均）にするとリークは減る。未実装。

2. **BPR との重複・ノイズ**  
   BPR 埋め込みはすでに「その映画をレビューした批評家たち」のパターンを低次元に押し込んでいる。  
   2-hop は同じ「その映画まわり」の情報を別形式（生の平均・人数）で足すので、**冗長なうえにノイズや多重共線性**で LGB が過適合しやすくなる可能性がある。

3. **train / test の分布の違い**  
   - 映画 M の「レビュワーたちの Fresh 率平均」は **train だけで**計算している。  
   - test の行では「同じ映画 M でも、別の批評家」がレビューしている。時系列で test が未来なら、**映画の評判やレビュワー層が変わっている**可能性があり、train で付けた 2-hop が test で外れやすくなる。  
   - レビュー数が少ない映画では、平均の**分散が大きく**、1 列の値が不安定になりやすい。

4. **bpr_count だけ微増だった理由**  
   `movie_review_count` は **target を使っていない**（`agg(..., movie_review_count=("target", "count"))` は行数カウント）。  
   リークがないので、BPR 16 + レビュー数 1 列では 0.76111 と微増。  
   → 2-hop で落ちる主因は **movie_fresh_rate_mean のリーク**と、上記の過学習・分布ずれの組み合わせと考えられる。

## 3. 1位解法との対応検証

### 3.1 割り算（run_atmacup_ratio）

| 1位の要件 | 実装 | 一致 |
|-----------|------|------|
| 集約前の値 / ユーザーごとの集約結果(平均) | 現在行の `base_feature` / 批評家ごとの `base_feature` の平均（train のみ） | ✅ |
| そのアイテムが好みからどれだけ離れているか | 比が 1 に近い＝傾向どおり、1 から離れる＝傾向から外れている | ✅ |
| 分母はユーザー(批評家)ごとの集約 | `groupby("critic_name")[base_feature].mean()` | ✅ |
| 未知ユーザー | `fillna(global_mean)` で分母を global 平均に | ✅ |
| 0 除算 | 分母 0 を `replace(0, np.nan).fillna(1.0)` で 1.0 に | ✅ |

**補足**: 批評家平均は train 全体で計算（時系列で「その行より前」だけにはしていない簡易版）。test は平均に使わないので test へのリークはない。

### 3.2 Implicit 埋め込み（run_atmacup_implicit）

| 1位の要件 | 実装 | 一致 |
|-----------|------|------|
| train, test の (user_id, item_id) ペアを結合 | train と test の (critic, movie) を両方行列に追加 | ✅ |
| 接続の有無だけ（評価値は使わない） | 行列の値はすべて 1。Fresh/Rotten は使っていない | ✅ |
| implicit で埋め込みを得る | ALS / BPR で fit、user_factors / item_factors を取得 | ✅ |
| 埋め込みを特徴量に | 批評家ベクトル + 映画ベクトルを 55 特徴に追加 | ✅ |
| 行列の形 | (users, items) = (n_critics, n_movies) の CSR。implicit の規約に準拠 | ✅ |

**協調フィルタリングの考え方（実装の流れ）**  
1. 批評家×映画の「誰がどの映画をレビューしたか」を 0/1 行列にする。  
2. ALS/BPR で低ランク分解し、批評家ベクトル・映画ベクトルを得る（似た批評家＝似たベクトル、似た映画＝似たベクトル）。  
3. 予測は内積で行わず、そのベクトルを**特徴量**として LGB に渡し、Fresh/Rotten を予測する。

## 4. 拡張の余地

| 軸 | 現状 | 拡張例 |
|----|------|--------|
| **割り算** | 1 特徴 1 列・批評家平均のみ | 複数特徴を同時に足す、**時系列で「その行より前」だけ**で批評家平均を取る、映画側平均で割る |
| **Implicit** | ALS/BPR・factors=16・ベクトル結合 | factors=8/32、**内積 1 列**のみ、confidence 重み、他手法 |
| **2-hop** | 未実装 | 「その映画をレビューした批評家」の統計を 1 列ずつ追加 |
| **Seen/Unseen** | 未実装 | 既知/未知の 2 モデル＋ブレンド |
| **組み合わせ** | 割り算と implicit は別提出 | 55＋割り算＋implicit を 1 本に載せる、スタッキングのメタ特徴に使う |

実装は `run_atmacup_ratio` / `run_atmacup_implicit` を土台に、パラメータ追加や別関数の追加で上記を載せていける形になっている。

### 4.1 行列分解の別手法（ALS/BPR 以外）

| 手法 | 考え方 | 備考 |
|------|--------|------|
| **ALS** | 「この批評家×この映画は 1 に近い／0 に近い」を**値**として当てる。二乗誤差を交互に最小化。 | ✅ 実装済み。0.75556。 |
| **BPR** | 「この批評家には、レビューした映画の方がしてない映画より上に来る」**順序**を当てる。ランキング学習。 | ✅ 実装済み。BPR 16 で 0.76101、**BPR 64 で 0.76479（現最高）**。 |
| **NMF** | 非負制約で分解。値の**非負の** latent の積で表現。ALS と異なる構造を捉える可能性。 | 未実装。`sklearn.decomposition.NMF` で同じ 0/1 行列を分解し、W (n_critics × k), H (k × n_movies) から批評家・映画ベクトルを得る。 |
| **TruncatedSVD** | 0/1 行列をそのまま SVD。特異値分解で低ランク近似。値の再構成誤差を最小化。 | 未実装。`sklearn.decomposition.TruncatedSVD` で同じ行列を分解。U が批評家、V が映画側の表現。 |
| **Logistic MF** | 相互作用を**確率**でモデル化（ロジスティック関数）。implicit 向けの論文あり。 | 未実装。`implicit` には標準でない。別ライブラリ（logistic-mf 等）か自前で試す候補。 |
| **内積 1 列** | 手法は ALS/BPR のまま。埋め込み 32 列ではなく **user_vec · item_vec の 1 列だけ**を特徴に追加。 | 未実装。CF の「この組み合わせのスコア」を直接 LGB に渡す。 |

以下を**明日試す**とよさそう。既存の `run_atmacup_implicit` やパイプラインを少し変えるだけで試せるものから並べる。

### 5.5 協調フィルタをさらに効果的に使うための実装の余地

§2.5 の結果（BPR 64 が最高、2-hop を足すと落ちる）を踏まえ、**協調フィルタをもっと効かせる**ために試せる実装を優先度・リスクで整理する。

#### 軸 1: BPR の次元・正則化（※試したが 0.76479 を超えず、次元増やしは得策ではなさそう）

軸1で BPR 128 / 256 / regularization / iterations を実施した結果、**いずれも 0.76479（BPR 64）を下回った**（上記「軸1 実験の Public 結果」参照）。256 が 0.76368 で軸1内最高だが、64 には届いていない。

| 案 | 内容 | 結果（Public） |
|----|------|----------------|
| **BPR factors=128 / 256** | 次元を 64 からさらに増やす。 | 128→0.76306、256→0.76368。**次元を増やすのはもう得策ではなさそう。** |
| **BPR の regularization** | 強くする(0.1)／弱くする(0.001)。 | reg_high→0.75609（落ちた）、reg_low→0.76292。 |
| **BPR の iterations 増加** | 200 に増やす。 | 0.76265。 |

→ **BPR は 64 次元で打ち止めとし、これ以上は「次元増やし」に頼らない**方針がよさそう。改良は軸2（ALS の次元・他分解）・軸4（2-hop 1 列）・軸5（ブレンド）に振る。

#### 軸 2: 別の行列分解を足す（足しすぎに注意）

| 案 | 内容 | 期待できること | 注意 |
|----|------|----------------|------|
| **ALS の factors 増加** | BPR が効いたので ALS も factors=32 / 64 を試す。 | BPR と目的関数が違うので、ALS 64 単体が効く可能性。 | 2-hop を足したときは落ちたので、**まずは BPR 64 か ALS 64 の単体**で効くか確認。 |
| **BPR 64 + ALS 32 の埋め込みを両方足す** | 55 + BPR 64 の 128 列 + ALS 32 の 64 列。 | 異なる分解なので相補的かもしれない。 | 列が一気に増えるので過学習リスク大。効いたらラッキー程度に試す。 |
| **NMF / TruncatedSVD** | 同じ 0/1 行列を NMF や SVD で分解し、批評家・映画ベクトルを 1 本で試す。 | BPR と異なる構造を捉える可能性。 | 未実装。1 本ずつ「NMF 16 のみ」「SVD 32 のみ」で効くか見る。 |

→ **「BPR が効いたから ALS の次元だけ増やす」は試す価値あり。BPR+ALS の同時足しはリスク高め。**

#### 軸 3: 行列の与え方・重み

| 案 | 内容 | 期待できること | 注意 |
|----|------|----------------|------|
| **confidence 重み** | 接続を 0/1 ではなく「レビュー数」や「時間の重み」で重み付ける。implicit の ALS は confidence をサポート。 | よくレビューされる映画・批評家の信号が強くなり、分解の質が変わる。 | BPR では仕様が違う場合あり。ALS で試すとよい。 |
| **時系列で行列を切る** | レビュー日で「過去だけ」の接続で行列を作り、将来を予測する形にする。 | リークが減り、汎化が良くなる可能性。 | 実装コストが高い。余裕があれば。 |

#### 軸 4: 2-hop は「使わない」か、ごく限定的に

| 案 | 内容 | 期待できること | 注意 |
|----|------|----------------|------|
| **2-hop は当面使わない** | 現状 2-hop を足すとほぼ落ちているので、**BPR の次元・正則化・ALS に注力する**。 | 足しすぎによる過学習を避けられる。 | 1 位解法では 2-hop が効いていたが、当コンペのデータでは効いていない。 |
| **2-hop のうち 1 列だけ（レビュー数）** | bpr_count は 0.76111 で BPR のみより微増していた。**BPR 64 + movie_review_count の 1 列だけ**を試す。 | 2-hop 3 列や割り算は落ちたが、レビュー数だけなら悪化を抑えられる可能性。 | あくまで 1 本試す程度。 |

#### 軸 5: ブレンド・アンサンブル

| 案 | 内容 | 期待できること | 注意 |
|----|------|----------------|------|
| **BPR 16 / 32 / 64 の予測をブレンド** | 単体では BPR 64 が最高だが、3 本の加重平均でさらに伸びるか試す。 | 異なる次元の補正で安定する可能性。 | 重みは CV や逆相関で決める。 |
| **BPR 64 と ALS 64 の予測をブレンド** | 2 本の提出の平均。 | 行列分解の違いが補正になる可能性。 | まずは各単体のスコアを出してから。 |

**実装の優先順位（提案・軸1結果を反映）**

- **BPR の次元増やし（128, 256）は実施済みで 0.76479 を超えなかったため、もう得策ではない。**
1. **ALS factors=64** を 1 本出す（既存 `run_atmacup_implicit(ctx, "als", 64, "als64")` で可）。  
2. **BPR 64 + movie_review_count の 1 列だけ**を 1 本試す（2-hop はこれだけ）。  
3. 効いた提出同士の**ブレンド**（BPR 64 と ALS 64 の平均など）を試す。  
4. 余裕があれば **confidence 重み**（ALS）や **NMF/SVD（CF 行列）** を 1 本ずつ試す。

2-hop をまとめて足す・割り算を足すのは当コンペではスコアを落としている。**BPR は 64 で打ち止め**とし、協調フィルタの拡張は **ALS・2-hop 1 列・ブレンド・重み/他分解** に振る方針がよさそう。

#### 5.6 現状の実装でまだ改良の余地があるか

**すでにやっていること（実装済み・実験済み）**

- BPR factors 16 / 32 / 64（および軸1で 128, 256, regularization, iterations）。**軸1の結果で 128/256 は 0.76479 を超えず、次元増やしは得策ではなさそう。**
- ALS factors=16（`run_atmacup_implicit(ctx, "als", 16, "als16")`）。ALS 32/64 は未実施。
- 行列は **0/1 のみ**（接続の有無だけ、重みなし）
- 2-hop 各種は試したが当コンペではスコア低下

**まだ実装していない／試し足りないこと（改良の余地）**

| 項目 | 現状 | やること | 実装コスト |
|------|------|----------|------------|
| **ALS の次元** | ALS 16 のみ実施 | ALS 32 / 64 を 1 本ずつ出す。`run_atmacup_implicit(ctx, "als", 64, "als64")` で可。 | 既存コードで実行するだけ |
| **confidence 重み** | 行列は常に 0/1 | 接続を「レビュー数」や時間重みで重み付けし、`scipy.sparse.csr_matrix((data, (rows, cols)), ...)` の `data` を変える。implicit の ALS は重み付き行列をサポート。 | `_build_implicit_embeddings` で行列構築部分を拡張（オプション引数で重み付きにする） |

| **NMF / TruncatedSVD（CF 行列）** | BPR/ALS のみ | 同じ (critic, movie) 0/1 行列を sklearn の NMF や TruncatedSVD で分解し、批評家・映画ベクトルを特徴として 1 本試す。 | 新規: 行列構築は流用し、分解だけ NMF/SVD に差し替える関数を追加 |
| **予測のブレンド** | 単体提出のみ | 例: BPR 64 と BPR 128（または ALS 64）の提出 CSV を読み、予測を加重平均して 1 本出す。 | `run_quick_embedding_submissions.py` の ensemble と同様のスクリプト or ノートブック 1 セル |

**結論（現状実装ベース）**:  
- **すぐ試せる**: ALS 64 の 1 本、BPR 64 + movie_review_count 1 列の 1 本、既存提出のブレンド。

- **コスト高**: 時系列で行列を切る。  

つまり、**協調フィルタとしてはまだ「ALS の次元・重み・他分解・ブレンド」を実装・実行する余地がある**。

**試さない方がよさそうなもの（現状の知見）**

- BPR の次元を 128 / 256 に増やす（軸1で 0.76479 未満だった）。
- 2-hop を 3 列まとめて足す、または Fresh 率平均・割り算を足す（スコアが大きく落ちた）。

## 6. Seen/Unseen（批評家基準）の実装案

### 6.0 1位解法の振り返りと「登場済み vs 未登場で精度が落ちる」への対応

**1位解法（atmaCup #15）の要点**

- **user（＝当コンペでは批評家）** で「train に登場した = seen」「登場していない = unseen」と分け、**seen 用・unseen 用の 2 モデル**を別々に学習。
- CV は **seen CV × 0.77 + unseen CV × 0.23** で見積もり。unseen の方が精度が落ちる前提で重み付け。
- 予測時は、test の各行が seen ユーザーなら seen 用モデル、unseen なら unseen 用モデルの出力を採用して 1 本の提出にまとめる。

**「登場済み vs 未登場で未登場の方が精度が落ちる」をモデルで分ける？ 協調フィルタでなんとかする？**

- **モデルで分ける**: 1位の通り、seen 用・unseen 用の 2 本を用意し、行ごとにどちらを使うか切り替える。**当コンペでは批評家の EDA の結果、test の批評家は 100% train に登場（unseen 批評家 = 0）** なので、**批評家基準の二モデルは不要**。分けてもいないし、分けたから精度が上がったわけでもない。
- **協調フィルタでなんとかする**: BPR は「train+test に登場する (批評家, 映画) の接続」で埋め込みを学習する。**批評家が unseen だと**、その批評家のベクトルは学習時に出てこない（または test で初出）ので、implicit の実装次第ではゼロベクトルなどになる。CF だけでは unseen 批評家の精度を「特別によくする」ことは難しく、1位のように **unseen 用は別モデル（批評家 ID 依存特徴を外す等）** にする方が現実的。
- **映画の「登場済み vs 未登場」**: 1位は **user** 基準で seen/unseen を分けている。**映画（item）側**で「train に登場した映画 vs 未登場の映画」を分けることは、1位では明示されていないが可能。当コンペでは **映画の seen/unseen はまだ EDA していない**。もし test に「train に 1 回も出てこなかった映画」があれば、その行では BPR の映画ベクトルはゼロ（または欠損）になり、精度が落ちやすい。その場合の選択肢は次のとおり。
  - **映画基準で二モデル**: 「その行の映画が train に登場済みか」で seen 映画用・unseen 映画用の 2 本を組み、行ごとに切り替える（1位の user 版の映画版）。
  - **協調フィルタだけでは解決しきれない**: unseen 映画は接続が train に無いので、BPR の映画ベクトルは学習されずゼロ等になる。CF は「接続がある」前提で効くので、unseen 映画は別扱い（映画 ID を落とした特徴だけのモデルや、global な映画特徴のみ）にする必要がある。

**現状の精度向上（0.76479）の理由**

- 上がった要因は **BPR の factors を 64 にしたこと**（協調フィルタの強化）であり、**seen/unseen をモデルで分けたことではない**。批評家の unseen が 0 なので、二モデル化は行っていない。

---

1位解法では **user（＝当コンペでは批評家）** で「train に登場した user = seen」「登場していない = unseen」とし、seen 用・unseen 用の 2 モデルを組み合わせている。当コンペでは **批評家基準** で同じことをする。

### 6.1 EDA（seen/unseen の割合）

- **スクリプト**: `scripts/eda_seen_unseen_critics.py`  
  - train/test の `critic_name` のみを読み（train はチャンク読みでメモリ節約）、次を集計する。  
  - 集計内容: train の行数・批評家ユニーク数、test の行数・批評家ユニーク数、**test の批評家のうち train に登場する数（seen）・登場しない数（unseen）**、**test の行数ベースで seen 批評家の行数・unseen 批評家の行数**。  
- **注意**: train.csv が約 730MB あるため、フル実行すると数分かかることがある。

**実行結果（当コンペ）**:

| 項目 | 値 |
|------|-----|
| Train | 653,507 行・1,573 ユニーク批評家 |
| Test | 40,716 行・1,483 ユニーク批評家 |
| Test の批評家のうち train に登場（seen） | 1,483（**100%**） |
| Test の批評家のうち train に未登場（unseen） | **0（0%）** |
| Test 行のうち seen 批評家 | 40,716（100%） |
| Test 行のうち unseen 批評家 | 0（0%） |

**結論**: 当コンペでは **test の全批評家が train に登場しており、unseen 批評家は 1 人もいない**。1 位の「Seen/Unseen 二モデル」をそのまま当コンペに適用する必要はなく、既知批評家用の 1 本のモデルで test 全体をカバーできる。

### 6.2 定義

- **Seen**: test のその行の `critic_name` が、train に 1 回以上登場している。  
- **Unseen**: test のその行の `critic_name` が、train に一度も登場していない。

### 6.3 実装の流れ（unseen が存在するコンペ向け）

当コンペでは unseen＝0 のため不要だが、**別コンペで test に未知批評家が含まれる場合**の参考として記載する。

| モデル | 学習データ | 特徴量 | 予測を出す対象 |
|--------|------------|--------|----------------|
| **既知批評家用** | train 全体 | 現行 55 特徴（`critic_name_te_ts` 等を含む） | test のうち「その行の批評家が seen」の行のみ |
| **未知批評家用** | train 全体 | 批評家 ID 依存を除く or global 置き換え（例: `critic_name` 削除、`critic_name_te_ts` を global 平均などで置換） | test のうち「その行の批評家が unseen」の行のみ |

- 学習: 既知用・未知用の 2 本の LGB をそれぞれ 1 回ずつ学習する。  
- 予測: test の各行について、その行の批評家が seen なら既知用モデルの予測、unseen なら未知用モデルの予測を採用する。  
- 提出: 上記で得た予測を 1 本の CSV（ID 順）にまとめて提出する。

unseen が存在するコンペでは、atmaCup #15 の「CV = seen×0.77 + unseen×0.23」のような**行数重み付き CV** を test の seen/unseen 割合に合わせて再現できる。
