# 7 パターン実験の方針と狙い

`train_7patterns_from_baseline.ipynb` で行う 7 パターンの**実験方針**と、**なぜその実験でスコアが伸びる可能性があるか**をまとめる。  
前提は `12_PUBLIC_SCORES_AND_NEW_BASELINE.md` の新ベースライン（0.76574 = BPR64 + BPR128 均等ブレンド）。

---

## 1. 全体方針

- **ベースライン**: 単体では BPR64（0.76479）と BPR128（0.76306）だが、**均等ブレンドで 0.76574** と単体を上回っている。
- **解釈**: 異なるモデル（次元数違い）の予測を合わせることで**分散が減り、過学習が和らぎ** Public が伸びたと考えられる。
- **方針**: 同じ土台（55 特徴 + BPR/ALS 埋め込み）の上で、
  - **ブレンドの重み**を変える（均等が最良とは限らない）
  - **3 本目（ALS64）**を足して多様性を増やす
  - **シード平均**で単一モデルの分散をさらに下げる
  - **2-hop の効いていた count1** をブレンドに組み込む  
  の 4 方向を、合計 7 パターンとして試す。

---

## 2. 各パターンの狙いと「上がりそう」な理由

### パターン 1: ベースライン再現（BPR64 + BPR128 均等）

| 項目 | 内容 |
|------|------|
| **やること** | 既存の 0.76574 を出すブレンドを同じコードで再計算する。 |
| **狙い** | 再現性の確認。ノートの 1 本目として「ここが基準」を明示する。 |
| **上がりそうな理由** | スコアを上げるためではなく、**検証用**。 |

---

### パターン 2: ブレンド重み 0.4 / 0.6（BPR64 軽め・BPR128 重め）

| 項目 | 内容 |
|------|------|
| **やること** | 予測 = 0.4 × BPR64 + 0.6 × BPR128。 |
| **狙い** | 単体では BPR64（0.76479）の方が BPR128（0.76306）よりわずかに高いが、**ブレンドで効いているのはどちらか**は分からない。BPR128 をやや重くして、次元を多くしたモデルの情報を強く出す。 |
| **上がりそうな理由** | 次元 128 の方が表現力が高い分、うまくいけば Public で効く可能性。均等より少し偏らせた重みで最適がずれているかもしれない。 |

---

### パターン 3: ブレンド重み 0.6 / 0.4（BPR64 重め・BPR128 軽め）

| 項目 | 内容 |
|------|------|
| **やること** | 予測 = 0.6 × BPR64 + 0.4 × BPR128。 |
| **狙い** | 単体スコアが良い BPR64 をやや重くし、過学習しやすい BPR128 の影響を弱める。 |
| **上がりそうな理由** | 単体で良い BPR64 の予測を主役にしたブレンドの方が、Public で安定する可能性。 |

---

### パターン 4: 3 本ブレンド均等（BPR64 + BPR128 + ALS64）

| 項目 | 内容 |
|------|------|
| **やること** | 予測 = (BPR64 + BPR128 + ALS64) / 3。 |
| **狙い** | ALS は BPR と**目的関数が違う**（ALS は二乗誤差、BPR はペアワイズ順序）ので、別の「見え方」の埋め込みになる。3 本にすることで**多様性が増し、分散削減**が期待できる。 |
| **上がりそうな理由** | 既存で BPR64+ALS64 が 0.76451 と高かった。ここに BPR128 を足すことで、BPR 同士の相関はあるが ALS が補正になり、ブレンド全体の安定化が期待できる。 |

---

### パターン 5: BPR64 シード平均（seed 42, 43, 44）

| 項目 | 内容 |
|------|------|
| **やること** | BPR64 を seed 42 / 43 / 44 で 3 本学習し、予測を均等平均して 1 本の提出にする。 |
| **狙い** | **同じモデル・同じ特徴**でシードだけ変え、予測のばらつきを平均で打ち消す。LGB や implicit の乱数の影響で出る分散を減らす。 |
| **上がりそうな理由** | アンサンブルの典型パターン。他コンペでも「シード平均で数値が少し伸びる」ことが多い。BPR64 単体 0.76479 をベースに、微増の可能性。 |

---

### パターン 6: bpr64_count1 + BPR128 均等ブレンド

| 項目 | 内容 |
|------|------|
| **やること** | 「BPR64 + 2-hop の movie_review_count 1 列」のモデル（bpr64_count1）と BPR128 を均等ブレンド。 |
| **狙い** | 単体では **bpr64_count1 が 0.76511** と BPR64 のみ（0.76479）より良い。2-hop の「その映画をレビューした人数」が効いている。これを BPR128 とブレンドし、**2-hop の利得とブレンドの分散削減**を同時に狙う。 |
| **上がりそうな理由** | count1 はリークのない 2-hop で、既に単体で効いている。ベースラインが「BPR64 + BPR128」なので、その BPR64 を「count1 付き」に差し替えたブレンドが、ベースライン 0.76574 を上回る可能性がある。 |

---

### パターン 7: 3 本ブレンド重み付き（BPR64 0.35, BPR128 0.45, ALS64 0.2）

| 項目 | 内容 |
|------|------|
| **やること** | 予測 = 0.35×BPR64 + 0.45×BPR128 + 0.2×ALS64。 |
| **狙い** | 単体スコアは BPR64 > BPR128 > ALS64 なので、**良い方に重みを多め**にしつつ、ALS で多様性を少し足す。均等 3 本より「BPR を主役、ALS は補助」にしたバランスを試す。 |
| **上がりそうな理由** | パターン 4 の均等 3 本だと ALS の影響が 1/3 と大きい。ALS 単体は 0.75867 とやや低いので、重みを下げて BPR を強くした方が Public で効く可能性。 |

## 4. コード上の確認済みポイント

- **ブレンド後の行順**: `reindex(ctx.test["ID"])` で提出用の ID 順に揃え、`save_submission(ctx.test, m["target"].values, ...)` で test と 1 対 1 対応になる。
- **欠損**: reindex で行が足りない場合に NaN が出るため、`fillna(0.5)` で補完してから保存（`save_submission` の sanitize でも NaN は 0.5 になるが、明示的に補完）。
- **verify_submission の path**: 返り値の `r["path"]` は文字列のため、`_report` では `Path(p).name` でファイル名のみ表示。
- **パターン 5**: seed 43/44 は `run_experiment(ctx_s, "bpr64_only_seed43", ...)` で `submission_2hop_bpr64_only_seed43.csv` に保存され、path 変数と一致。

以上を方針として、ノート実行後は Public スコアでどれが効いたかを 12 のドキュメントに追記していく想定。
