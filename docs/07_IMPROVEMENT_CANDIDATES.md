# 劇的な精度改善の候補（train_baseline 追加要素）

`train_baseline.ipynb` を土台に、**本番ベースライン 0.75493 を大きく超える**ための候補を、ドキュメント（01〜06, PROJECT_MIND, BASELINE_VERIFICATION）を参照して整理する。  
「追加要素」＝ベースラインの**上に載せる**実験・特徴・パイプライン変更。

---

## 前提の整理

| 項目 | 内容 |
|------|------|
| 現状ベースライン | 38 特徴のみ → submission.csv（メタのみ）。本番で超える目標は **0.75493**（embedding + PCA16 + doc_x_critic_te）。 |
| 改善の土台 | 実質的には **0.75493 を出すパイプライン**（quick_embedding の doc_x_critic_te）を拡張する形が効きやすい。train_baseline は「メタのみ」の比較用・分析用として維持。 |
| ドキュメントの教訓 | 特徴量の**大量追加は Public で落ちた**（02 段階的 9 パターン全ボツ）。**1〜2 列に絞る**か、**別軸（embedding・アンサンブル・疑似ラベル）**で攻める。 |

---

## 候補一覧（優先度・期待度・コスト）

### 1. 疑似ラベル（Pseudo-Labeling）【期待度: 高 / コスト: 中】

**内容**  
- 本番ベースライン（0.75493 のパイプライン）で一度 test を予測する。  
- 予測確率が **0.95 以上** または **0.05 以下** の行だけを擬似正解として train に追加し、再学習して test を再度予測。

**根拠**  
- PROJECT_MIND §8.1: 「文脈が効く映画レビューと相性が良く、試す価値が高い」「実装優先度: 高」。  
- 同じデータ・同じモデル族で「自信の高い予測」を教師に使うだけなので、train_baseline の延長や quick_embedding の後段として組み込みやすい。

**train_baseline との関係**  
- ベースラインで「全 train で 1 本学習 → test 予測」したあと、**追加セル**で「高確信度のみ擬似ラベル化 → 再学習 → 再予測」を実行する形で追加できる。

**注意**  
- 閾値は厳しめ（0.95/0.05）から。不均衡（Fresh ばかりなど）にならないか確認し、必要ならサンプリングや重みで調整（PROJECT_MIND §8.1）。

---

### 2. 類似度ベース TE（似た批評家 / 似た映画だけの TE）【期待度: 高 / コスト: 高】

**内容**  
- 通常の TE ではなく、「**その批評家に似た批評家 top-k**」がその映画に付けた Fresh 率、「**その映画に似た映画 top-k**」にその批評家が付けた Fresh 率を特徴量にする。  
- 類似度は **Implicit（正解ラベルを使わない）** が無難: 批評家ごと「レビューした映画の集合」→ 批評家間 Jaccard。映画ごと「レビューした批評家の集合」→ 映画間 Jaccard。

**根拠**  
- PROJECT_MIND §7 (G) atmaCup #15 6位: 「似ていないユーザー・作品も平均に混ざる」のを避け、**似た者だけ**で TE を取ると Private で大きく伸びた。  
- 手順は PROJECT_MIND §7.2 (1) に詳細あり（時系列を守る場合の「その行より前」のみで集合を作る等）。

**train_baseline との関係**  
- 38 特徴に加えて「似た批評家 top-k のその映画の Fresh 率」「似た映画 top-k のその批評家の Fresh 率」を **1〜2 列ずつ**（k=5, 10 など少数）追加するセルを足す。  
- 既存の `get_baseline_data()` の上に、**批評家・映画の集合と Jaccard → top-k TE** を計算するブロックを追加する形。

**注意**  
- 実装・計算コストは大きい。まず **Jaccard のみ・k を 1〜2 個**に絞って CV が伸びるか確認するのが現実的（PROJECT_MIND §7 (G) 末尾）。

---

### 3. 多モデル + Ridge スタッキング（LGB・XGB・CatBoost → Ridge）【期待度: 中〜高 / コスト: 中】

**内容**  
- **同じ 38 特徴**（または 0.75493 パイプラインと同じ特徴）で、LightGBM・XGBoost・CatBoost を同じ時系列 CV で学習。  
- 各 fold の val で 3 モデルの予測を **メタ特徴** とし、Ridge でメタ学習。  
- 最終提出は「各 fold の 3 モデル → Ridge で 1 本に → fold 平均」で test を予測。

**根拠**  
- PROJECT_MIND §7 (I) atmaCup #15 2位: 同一特徴で LGB/XGB/CatBoost を組み、Ridge スタッキングで CV が 1.12→1.09→1.085 と段階改善。  
- 手順は PROJECT_MIND §7.2 (5) に詳細あり（メタ学習に val の正解を使う、test は使わない）。

**train_baseline との関係**  
- 現在は LGB のみなので、**「LGB のブロックの下」に XGB・CatBoost を足し、Ridge メタと OOF/test 予測のブロック**を追加する形。  
- まずは **メタのみ 38 特徴**で試し、効けば **0.75493 パイプライン（embedding+PCA16+doc_x_critic_te）** でも同様に 3 モデル + Ridge を試す。

---

### 4. 次元数バリエーション + 全データ 1 本のアンサンブル【期待度: 中 / コスト: 低】

**内容**  
- 同じ特徴（例: 38 + movie_title_info + PCA）で、**PCA の n_components を 8, 16, 32** などに変えた複数本を学習。  
- さらに「**全 train で 1 本**（early_stopping なし）」も 1 本作る。  
- 提出は「CV の fold 平均」複数本 + 「全データ 1 本」を **均等平均または重み付き平均** でブレンド。

**根拠**  
- PROJECT_MIND §7 (F) atmaCup #15 3位: 「全データ 1 本を足すと Public が上がった」と明言。§7.2 (2) に手順あり。  
- 05 では seed_avg（0.75483）、ensemble_two（0.75478）、pca32（0.75465）など、単体では 0.75493 に届かないが、**ブレンドで効く可能性**がある。

**train_baseline との関係**  
- メタのみ 38 特徴では「次元数」のバリエーションはないので、主に **embedding パイプライン側**（quick_embedding や train_embedding_submission）で実施。  
- train_baseline としては「**複数シードで学習した 1 本**」と「**全データ 1 本**」をブレンドする**提出用セル**を 1 つ足すイメージ（embedding 側で作った CSV を読み、ブレンドして保存するだけでも可）。

---

### 5. Rotten 重視の学習（scale_pos_weight / class_weight）【期待度: 中 / コスト: 低】

**内容**  
- LightGBM で `scale_pos_weight = (Rotten 件数) / (Fresh 件数)` を指定する。  
- または `class_weight="balanced"` を計算し `sample_weight` で渡す。  
- 現状「Fresh 出しすぎ」（FP 多め）なので、Rotten を大事にすることで閾値まわりの改善を狙う。

**根拠**  
- 03_ANALYSIS §5、04_METRICS_GUIDE: 弱点セグメント・FP/FN の話と整合。Rotten を外している箇所を減らすことが AUC 改善に直結しうる。

**train_baseline との関係**  
- `LGBMClassifier(..., scale_pos_weight=...)` を 1 本追加するだけ。時系列 CV と「全 train で 1 本」の両方で同じ重みを使い、CV と提出を比較する。

---

### 6. CV スコアの加重（時系列 AUC と GroupKFold AUC）【期待度: 中 / コスト: 低】

**内容**  
- テストに**未知映画**が一定割合あると仮定し、**0.7×時系列CV_AUC + 0.3×GroupKFold_AUC** のような 1 本の CV 指標でモデル比較する。  
- 学習手順は変えず、**評価指標だけ**を変える。

**根拠**  
- PROJECT_MIND §7 (H) 5位、§7.2 (3): テストの seen/unseen 比率に合わせて CV を重み付ける。

**train_baseline との関係**  
- 時系列 CV は既にある。**GroupKFold（group=映画）** を追加で回し、両方の AUC を出してから加重平均を計算するセルを足す。  
- モデル選択や early_stopping を「この加重 AUC」で行うようにすると、本番に近い選び方ができる。

---

### 7. 特徴選択（全 fold で重要度 0 の列だけ除去）【期待度: 低〜中 / コスト: 低】

**内容**  
- 時系列 CV の**各 fold** で LightGBM の feature_importances_ を取得。  
- **全 fold で重要度 0** だった列だけ特徴量から除外し、その特徴量で再 CV・再提出する。

**根拠**  
- PROJECT_MIND §7 (I) 2位、§7.2 (4): ノイズ列削減で過学習抑制。  
- 03 では 0 重要度の列が多く出ているので、削ってみる価値はある。

**train_baseline との関係**  
- 既存の「重要度表示」のあとに、「全 fold で 0 の列リストを作る」セルと、「その列を除いた FEATURES で再学習」セルを足す。

---

### 8. テキストの「もう 1 ビュー」: TF-IDF + SVD または BERT 系【期待度: 中 / コスト: 高】

**内容**  
- **TF-IDF + TruncatedSVD**: movie_info や タイトル+映画情報を TF-IDF → SVD で 20〜50 次元に圧縮し、**OpenAI embedding とは別の列**として追加。embedding と共存させる。  
- **BERT / RoBERTa**: 同じテキストを BERT や感情分析用 RoBERTa でエンコードし、映画単位のベクトルを 1 本追加。OpenAI と concat してから GBDT。

**根拠**  
- PROJECT_MIND §7 (A)(B)(D)、§8.3: TF-IDF+SVD は「embedding と共存」がまだ本格的でない。BERT は「締切直前の追加ブースト候補」。  
- 01: TF-IDF+SVD は微増だったが、**embedding と併用**は未検証。

**train_baseline との関係**  
- メタのみ 38 特徴のベースラインに「TF-IDF-SVD の 20 列」を足す 1 設定を追加する、あるいは **0.75493 パイプライン**に TF-IDF-SVD 列を足す。  
- BERT は計算・実装コストが大きいので、TF-IDF-SVD を先に試すのが現実的。

---

### 9. グラフ埋め込み（批評家–映画二部グラフで ProNE / Node2Vec）【期待度: 中 / コスト: 高】

**内容**  
- 「批評家–映画」の「レビューした」二部グラフを組み、ProNE や Node2Vec で批評家ベクトル・映画ベクトルを学習。  
- そのベクトル（または類似度）を数列だけ特徴量に追加する。

**根拠**  
- PROJECT_MIND §7 (H) 5位: user–anime 二部グラフで ProNE が効いている。批評家–映画で同様の構成が可能。

**train_baseline との関係**  
- 38 特徴（または 0.75493 パイプライン）の**上に**、グラフから作った数列を足すブロックを追加。  
- 実装が重いので、類似度 TE（候補 2）やスタッキング（候補 3）のあとに回す候補。

---

### 10. 弱点セグメント向けの追加 1 列（2015 年・top_critic）【期待度: 低〜中 / コスト: 低】

**内容**  
- 既に **doc_x_critic_te**（Documentary × critic_te）で 0.75493 まで伸びている。  
- 同様に **review_year が 2015 のときだけ効く列**や **top_critic × critic_te** を **1 列だけ**追加して CV を確認。  
- 02 では topcritic_x_critic_te は base と同一だったが、embedding ありのときは効く可能性がある。

**根拠**  
- 03_ANALYSIS: 弱点は 2015 年、Documentary、top_critic。04: doc_auc と 2015 / top_critic を併せて見る。  
- 過学習を避けるため、**1 列に絞り**、CV と Public の両方で確認する（02 の教訓）。

**train_baseline との関係**  
- train_metric_driven_experiments で既に year_norm_x_critic_te, topcritic_x_critic_te を試している。  
- **embedding パイプライン**（0.75493 の土台）に、それらを **1 列ずつ**足した設定を 1 本ずつ提出して検証する、という位置づけ。

## 推奨する試す順序（劇的な改善を狙う場合）

1. **疑似ラベル**（候補 1）— ドキュメントで優先度・相性ともに高く、パイプライン拡張がしやすい。  
2. **Rotten 重視**（候補 5）— 実装が軽く、弱点（FP 多め）に直接効かせられる。  
3. **多モデル + Ridge スタッキング**（候補 3）— 同じ特徴で伸ばす定番。  
4. **次元数バリエーション + 全データ 1 本アンサンブル**（候補 4）— 既存提出のブレンドで試せる。  
5. **類似度 TE**（候補 2）— 効果は大きいが実装コスト高。余力があれば。  
6. **CV 加重**（候補 6）・**特徴選択**（候補 7）— 評価と特徴の整理として常時利用可能。  
7. **TF-IDF-SVD の追加ビュー**（候補 8）— embedding と併用がまだなので 1 本試す価値あり。  
8. **2015 / top_critic の 1 列**（候補 10）— 0.75493 パイプラインに 1 列足しての検証。

| **10** year_norm_x_critic_te | ✅ P15 と同一（02 §3.1）。train の min/max で正規化、リークなし。 | CV +0.0006 の実績あり。Public は微増〜横ばいの可能性。 |

### 提出スコアがベースラインより低い場合の解釈

- **07（特徴選択）**: 0 列除去のときは **38 特徴・全データ 1 本**でベースラインと同一。Public **0.75097** は「38 特徴のみ」のベースラインそのもの（PROJECT_MIND の「ベースライン38のみ」と一致）。実装ミスではない。
- **06（GroupKFold）**: 4 本とも「全データの約 75% で学習」したモデルの予測を平均しているため、**全データ 1 本**より Public が下がる設計。0.74 台は想定内。
- **05（scale_pos_weight）**: 最適化が AUC から少しずれるため、AUC がわずかに下がることがある。
- **10（1 列追加）**: 過去の教訓どおり、1 列追加で Public が微減した可能性（過学習・ノイズ）。

### 55 特徴ベース（0.75493）で出し直しても軒並みベース以下だった理由の考察

**結果の整理**（ベース 0.75493 に対して）  
07: 0.75450（やや近い）／08: 0.75386／01: 0.75375／10: 0.75361／05: 0.75346／06: 0.74312（設計上低め）。

**全体としての理由**  
- ベース（doc_x_critic_te の 1 本）がすでに「55 特徴・全データ 1 本」の単純な最良構成に近く、**余計な操作がノイズになる**構造になっている。  
- **scale_pos_weight（05）**: 目的が AUC からずれるので、そのままでは Public AUC が落ちやすい。  
- **1 列追加（10）**: 過去と同様、列追加で過学習・分布ずれで微減しやすい。  
- **TF-IDF（08）**: 既に embedding でテキスト情報は入っており、**二重に効かせている**か、SVD のノイズで悪化している可能性。  
- **特徴選択（07）**: 0 列除去ならベースと同一になるが、何列か落とすと「有用な情報を削っている」可能性。0.75450 は 1 本に近いが、除去した列がわずかに効いていた可能性。  
- **GroupKFold（06）**: 4 本とも約 75% で学習したモデルの平均なので、**設計上ベースより低くなりやすい**（0.74 台は想定内）。

**01 疑似ラベルがベースより低いことの解釈（「自信満々＝正しい」とは限らない）**  
- 0.95/0.05 で「かなり自信がある」と判断した test だけを擬似ラベルにして再学習しているのに 0.75375 とベースを下回っている。  
- これは **「自信満々でつけたラベルが、実は結構外れている」** 可能性を強く示す。  
  - モデルが **過信（overconfident）** しており、確率 0.95 付近の予測の正答率が 0.95 よりかなり低い（キャリブレーションが悪い）。  
  - その「外れた自信高い予測」を教師として足すと、**誤ったパターンを強化**してしまい、全体の AUC が落ちる。  
- つまり「そこそこ自信満々につけてるやつが意外と正しくない」という解釈でよい。  
- 対策の方向性: **擬似ラベルをさらに少なく・厳しくする**（閾値 0.98/0.02、または件数キャップをきつく）、**擬似ラベルの重みをさらに下げる**、あるいは **キャリブレーションを改善してから**擬似ラベルを採る（プラットスケーリング等）。

### 実装を少し改善したらスコアが上がりそうな考察（コードは未変更・方針メモ）

- **05 scale_pos_weight**: 現状は `n_neg/n_pos` で不均衡重みを付けており、最適化が AUC からずれて Public が微減しうる。**class_weight の強さを弱める**（例: 0.5 * n_neg/n_pos）や、**AUC をメトリクスにした early_stopping を維持したまま sample_weight だけ付ける**などにすると、Rotten を効かせつつ AUC の落ち込みを抑えられる可能性がある。
- **06 GroupKFold**: 現状は「4 本の約 75% 学習モデルの平均」で提出しているため Public が落ちる。**提出用に「全データで 1 本再学習」を別ファイルで出す**オプションは、38 特徴のままなら**中身はベースライン（submission.csv）と同一**になる。06 の本質は **CV（GroupKFold）で未知映画への汎化を評価すること**であり、提出がベースライン並み＝ベースラインそのものを 06 の run からも出せる、という意味。
- **07 特徴選択**: 0 列除去だけだと 38 のままなので変化なし。**重要度下位 N 列を捨てる**や **L1 正則化で自動選択**を試すと、ノイズ削減で Public が微増する可能性がある（過学習抑制）。ただし 02 の教訓で「列を減らしすぎると悪化」もあるので N は小さめから。
- **10 1 列追加**: year_norm_x_critic_te 単体では Public 微減だった。**doc_x_critic_te（0.75493 の軸）に 1 列足す**形で試すと、既に効いている軸の上なのでスコアが乗りやすい可能性がある。列追加は「embedding パイプライン上で 1 列」が無難。
- **01 疑似ラベル**: 閾値 0.95/0.05 を**少し緩める（0.90/0.10）**と擬似ラベル数が増えて効果が出やすくなる一方、ノイズも増える。**擬似ラベル数を上限でキャップする**（例: 最大 2000 件）と過学習を抑えつつスコアが伸びる可能性がある。
- **04 ブレンド**: 均等平均ではなく **CV や過去提出の相関に基づく重み**（例: 逆相関で重み付け）を入れると、同じ本数でもスコアが少し伸びる可能性がある。

**まとめ**: 実装は意図どおり。**0.75493 を超える提出を狙うなら**、(1) **04 で doc_x_critic_te を含むブレンド**を出す、(2) 05・01・03 などを **quick_embedding の doc_x_critic_te パイプラインに載せて**提出、のいずれかが有効。

## 次に試すこと（55 特徴ベースの結果を受けて）

「単体の改善候補は軒並みベース以下だった」ことを踏まえた、**次の一手の候補**。  
**特徴量を積み上げるだけが答えではない**。同じ 55 特徴のまま「予測の出し方」を変える方が効く可能性が高い。

### 優先度高（特徴量を増やさない）

| 案 | 内容 | コスト・注意 |
|----|------|----------------|
| **04 ブレンド（重み付き）** | doc_x_critic_te（0.75493）を必ず含め、既存の複数 CSV を**均等でなく重み付き平均**（例: 逆相関・CV スコアで重み）。 | 低。既存 CSV があればすぐ試せる。 |
| **シード平均** | 55 特徴・同じパイプラインで **seed 42, 43, 44 など複数本学習 → 予測を平均**。quick_embedding の seed_avg は 0.75483 の実績あり。 | 低。学習数回するだけ。 |
| **PCA 次元だけ変えたブレンド** | pca16（doc_x_critic_te）, pca24, pca32 など**同じ embedding で次元だけ変えた複数本**を 1 本にブレンド。特徴列は増やさない。 | 低。quick_embedding で既に複数本あるならその平均。 |
| **03 スタッキング** | **同じ 55 特徴**で LGB・XGB・CatBoost を学習 → Ridge でメタ。特徴は積み上げず「モデルの多様性」で伸ばす。 | 中。XGB/CatBoost の導入が必要。 |

### 特徴量は「慎重に 1 列ずつ」

- 過去の教訓: **列をたくさん足すと Public が落ちた**。やるなら **1 列に絞り、CV で効いているか必ず確認**してから提出。
- 候補: **02 類似度 TE**（55 特徴ベースではまだ未提出。Jaccard で似た批評家/映画の Fresh 率 1〜2 列）や、**既に CV で効いていた別軸の 1 列**（publisher_freq 等）を **55 特徴に 1 列だけ足した 1 本**を出す。
- **year_norm_x_critic_te（10）は 55 特徴に足したら 0.75361** だったので、別の 1 列を試すか、列追加は見送るか判断材料になる。

### 擬似ラベルを再挑戦する場合

- 01 は「自信高い＝正しい」とは限らないことが分かった。再挑戦するなら: **閾値を 0.98/0.02 に厳しくする**か**件数キャップをさらに減らす**、**擬似ラベルの sample_weight を 0.2 以下に下げる**。  
- あるいは **プラットスケーリング等でキャリブレーションを改善してから**、改めて高確信のみを擬似ラベルにする。

### やらない方がよさそうなこと（現時点）

- **55 特徴に TF-IDF をさらに足す（08）**: 0.75386。embedding と二重になりがち。  
- **55 特徴に year_norm_x_critic_te を足す（10）**: 0.75361。1 列追加で過学習しやすい。  
- **scale_pos_weight をそのまま単体で提出（05）**: AUC がずれて 0.75346。目的を変えたいときだけ。

### まとめ

- **まずは「特徴量を積み上げずに」**: 04 ブレンド（重み付き）・シード平均・PCA 次元バリエーションのブレンド・03 スタッキングで、**同じ 55 特徴のまま予測の出し方だけ変える**のが現実的。  
- **特徴量積み上げは「必要なら 1 列ずつ・CV で効くか確認」**。02 類似度 TE を 55 特徴に 1 列足して 1 本出すのは試す価値あり。  
- これでも伸びなければ、embedding の質（モデル・前処理）や時系列の切り方など、**土台そのものの見直し**に進む段階。

## atmaCup #15 1位解法を踏まえた「一つの手法を深める」5 提出案

**参照**: atmaCup #15 1st place solution（seen/unseen 分割・集約特徴・「割り算」・implicit 埋め込み・GNN 等）。  
当コンペでは **批評家 = user、映画 = item、「レビューした」= 接続** に対応づける。今日の残り 5 提出を「一つの軸を深める」形で使う場合の 5 パターン案（実装は別途）。

### 1. 批評家ごと集約 + 「値 / 批評家平均」（割り算特徴）

**1位解法の要点**: 「集約前の値 / ユーザーごとの集約結果(平均)」が特に良かった。そのアイテムが好みからどれだけ離れているかを表す。LGB は差より割り算が効く。

**当コンペでの具体**  
- 批評家ごとに、その批評家の train 内レビューで **ある数値特徴の平均** を計算（例: `genre_Documentary` の平均、`critic_name_te_ts` の平均）。  
- 特徴量: **現在行の値 / その批評家の平均**（0 除算は 1 や global 平均で置換）。  
- **5 提出パターン**: 分母に使う「批評家ごと集約」を 5 種類に分けて **1 列ずつ** 55 特徴に足し、それぞれ 1 本ずつ提出。  
  - 例: (1) `genre_Documentary / critic_mean_genre_Documentary`  
  - (2) `critic_name_te_ts / critic_mean_critic_name_te_ts`  
  - (3) `runtime_bin / critic_mean_runtime_bin`  
  - (4) `movie_age_bin / critic_mean_movie_age_bin`  
  - (5) `release_decade / critic_mean_release_decade`  
- 未知批評家は global 平均で割るか、その列を欠損にして LGB に任せる。

### 2. Implicit 埋め込み（接続情報のみの協調フィルタリング）

**1位解法の要点**: test にも「視聴した」という接続がある。train+test の (user_id, anime_id) を結合し、implicit で埋め込みを得て特徴量に。

**当コンペでの具体**  
- (critic, movie) の「レビューした」接続は train にも test にもある（test はラベルなしだけ）。  
- **implicit**（ALS / BPR 等）で train+test の (critic_id, movie_id) ペアから **critic 埋め込み・movie 埋め込み** を学習し、その行の critic_vec と movie_vec を結合（または内積 1 列）して 55 特徴に足す。  
- **5 提出パターン**: 次元数や手法を変えて 5 本。  
  - 例: (1) ALS factors=8  
  - (2) ALS factors=16  
  - (3) ALS factors=32  
  - (4) BPR factors=16  
  - (5) 内積のみ 1 列（ALS 16 の critic・movie を内積で 1 列にまとめる）

### 3. Seen / Unseen 二モデル（既知批評家用・未知批評家用）

**1位解法の要点**: seen 用と unseen 用で別モデル。CV は seen CV × 0.77 + unseen CV × 0.23 で見積もる。

**当コンペでの具体**  
- 既に「既知/未知 2 本」で 0.75079 の実績あり（PROJECT_MIND）。  
- **5 提出パターン**: 二モデルの **ブレンド重み** を変えて 5 本出す。  
  - 既知批評家の test 行にはモデル A、未知批評家の test 行にはモデル B の予測を使い、その「混合率」は固定ではなく、**提出ごとに「A の重み」を 0.5, 0.6, 0.7, 0.8, 0.9 のように変えた 5 本**（または A:B を 5:5, 6:4, 7:3, 8:2, 9:1）。  
- あるいは、seen/unseen の **定義**（例: レビュー数 N 本以上を seen）を 5 パターンに変えて 5 本、でもよい。

### 4. 2-hop 集約（「その映画をレビューした批評家」の統計）

**1位解法の要点**: user ごと集約 → そのアイテムに対する「そのアイテムを視聴したユーザーたち」の集約をさらに集約して 2-hop の特徴に。

**当コンペでの具体**  
- 映画 m について、「m をレビューした批評家たち」の **Fresh 率の平均** や **その批評家の critic_name_te_ts の平均** を 1 列として追加（1-hop は既存 TE、2-hop は「その映画に付いた批評家群」の統計）。  
- **5 提出パターン**: 2-hop で何を集約するかを 5 種類に分けて 1 列ずつ足し、5 本。  
  - 例: (1) その映画をレビューした批評家の train 内 Fresh 率の平均  
  - (2) その映画をレビューした批評家の `critic_name_te_ts` の平均  
  - (3) その映画をレビューした批評家の `genre_Documentary` の平均  
  - (4) その映画をレビューした批評家の人数（カウント）  
  - (5) その批評家がレビューした映画の `genre_Documentary` の平均（批評家側 1-hop のさらに映画側集約）

### 5. 割り算 + 2-hop の組み合わせ（1 列ずつ 5 本）

**1位解法の要点**: 集約と割り算の組み合わせが効いている。

**当コンペでの具体**  
- 上記 1 と 4 を組み合わせる。  
- **5 提出パターン**: 次のいずれかを **1 列だけ** 55 特徴に足して 1 本ずつ提出。  
  - (1) 批評家平均で割る: `genre_Documentary / critic_mean_genre_Documentary`  
  - (2) 批評家平均で割る: `critic_name_te_ts / critic_mean_critic_name_te_ts`  
  - (3) 2-hop: その映画をレビューした批評家の Fresh 率の平均  
  - (4) 2-hop: その映画をレビューした批評家の `critic_name_te_ts` の平均  
  - (5) 割り算と 2-hop の交互作用っぽく: 2-hop Fresh 率 / その批評家の train 内 Fresh 率  

---

**今日の 5 提出の使い方の目安**  
- **一つの手法に絞る**: 上記の **1（割り算）** か **2（implicit 埋め込み）** のどちらかで、**5 パターン = 5 本** が作りやすい。  
- 1 は既存 55 特徴＋集約計算だけなので実装が軽く、2 は `implicit` ライブラリで BPR/ALS を回すだけなので、どちらも「一つの手法を深める」のに向いている。  
- 03 スタッキング 5 本（seed 違い）はコンペ終盤用に残し、今日は上記 1 または 2 で 5 本を揃える、という割り振りが現実的。

### 1・2 は今までに試していたか

| 案 | 試したか | 補足 |
|----|----------|------|
| **1. 批評家ごと集約 + 割り算** | **一部のみ** | **v2_ratio**（`archive/top_solutions.py`）で **TE 系の割り算**は試済み: `movie_te / critic_avg_fresh`（批評家の **Fresh 率の平均**で割る）、`movie_te / global`、`critic_te / global`。v2 + ratio は 0.73〜0.74 台でベースラインより悪化（PROJECT_MIND）。**atmaCup 1 位で効いた「その特徴の値 / ユーザー(批評家)のその特徴の平均」**（例: `genre_Documentary / critic_mean_genre_Documentary`）は **未試**。土台が v2 だったので、**55 特徴ベースで「特徴量ごとの批評家平均で割る」1 列**はまだやっていない。 |
| **2. Implicit 埋め込み** | **未試** | 批評家×映画の **NMF 分解**（`top_solutions.py` の add_nmf_features）は試しているが、**implicit ライブラリの ALS/BPR** は使っていない。接続 (critic, movie) を train+test で結合して implicit で埋め込みを得る、という 1 位のやり方そのものは **未実施**。 |

→ **1 の「特徴量ごとの批評家平均で割る」** と **2 の implicit ALS/BPR 埋め込み** は、どちらもまだ試していない。

### 1位解法との実装対応の検証（lib/improvement_candidates.py）

**1. 割り算特徴（run_atmacup_ratio）**

| 1位の要件 | 実装 | 一致 |
|-----------|------|------|
| 集約前の値 / ユーザーごとの集約結果(平均) | 現在行の `base_feature` / 批評家ごとの `base_feature` の平均（train のみ） | ✅ |
| そのアイテムが好みからどれだけ離れているか | 値が 1 に近い＝批評家の傾向と同程度、1 から離れる＝傾向から外れている | ✅ |
| 分母はユーザー(批評家)ごとの集約 | `groupby("critic_name")[base_feature].mean()` で批評家ごと平均 | ✅ |
| 未知ユーザー | 批評家が train にいない場合は `fillna(global_mean)` で分母を global 平均に | ✅ |
| 0 除算 | 分母 0 を `replace(0, np.nan).fillna(1.0)` で 1.0 に置換 | ✅ |

**補足**: 批評家平均は **train 全体**で計算している（時系列で「その行より前」だけにしていない簡易版）。test は平均に一切使っていないので test へのリークはない。

**2. Implicit 埋め込み（run_atmacup_implicit）**

| 1位の要件 | 実装 | 一致 |
|-----------|------|------|
| train, test の (user_id, item_id) ペアを結合 | train の (critic, movie) と test の (critic, movie) を両方 `rows, cols` に追加 | ✅ |
| 接続の有無だけ（評価値は使わない） | 行列の値はすべて 1（`np.ones`）。Fresh/Rotten は使っていない | ✅ |
| implicit で埋め込みを得る | `implicit.als.AlternatingLeastSquares` / `implicit.bpr.BayesianPersonalizedRanking` で fit | ✅ |
| 埋め込みを特徴量に | `user_factors`（批評家ベクトル）と `item_factors`（映画ベクトル）を各行に付与し、55 特徴に追加 | ✅ |
| 行列の形 | (users, items) = (n_critics, n_movies) の CSR。implicit の fit(user_items) の規約に準拠 | ✅ |

**結論**: 1 位解法の「割り算」と「implicit 埋め込み」の考え方は、上記のとおり実装に反映されている。

### ここから拡張する余地

現状は「5 パターン・1 列ずつ／1 手法ずつ」に絞っているので、以下のように拡張余地がある。

| 軸 | 現状 | 拡張例 |
|----|------|--------|
| **割り算** | 1 特徴につき 1 列・批評家平均のみ | 複数特徴の割り算を同時に足す、**時系列で「その行より前」だけ**で批評家平均を取る、映画側平均で割るバリエーション |
| **Implicit** | ALS/BPR・factors=16・埋め込みをそのまま結合 | factors=8/32、**内積 1 列**のみ、別の重み（confidence）を付ける、implicit の他手法 |
| **2-hop** | 未実装 | 「その映画をレビューした批評家」の統計を 1 列ずつ追加（07 提出案の 4） |
| **Seen/Unseen** | 未実装 | 既知批評家用・未知批評家用の 2 モデル＋ブレンド（07 提出案の 3） |
| **組み合わせ** | 割り算と implicit は別提出 | 55＋割り算 1 列＋implicit を同じ 1 本に載せる、スタッキングのメタ特徴に使う |

実装は `run_atmacup_ratio` / `run_atmacup_implicit` を土台に、**パラメータ追加や別関数の追加**で上記を載せていける形になっている。
